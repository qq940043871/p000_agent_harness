# 06 大道至简：解密 OpenClaw 最简工具集法则与 YOLO 执行哲学

## 🎯 本节目标

理解"最简工具集"的设计哲学，掌握 YOLO 执行哲学，知道什么时候应该少做而不是多做。

---

## 1. 工具集膨胀的陷阱

大多数 Agent 项目都会犯同一个错误：**工具越加越多**。

```
v0.1：bash, read_file, write_file         ✓ 够用
v0.2：+ search_web, send_email            ✓ 合理扩展
v0.3：+ calendar, database, translate...  ⚠ 开始膨胀
v0.4：+ 50 个专用工具...                  ✗ 工具噪声问题
```

### 工具过多的代价

| 问题 | 说明 |
|---|---|
| **Prompt 膨胀** | 工具描述占用大量 Context 窗口 |
| **决策混乱** | LLM 在相似工具间反复纠结 |
| **维护成本** | 每个工具都需要测试和文档 |
| **幻觉风险** | 工具太多，LLM 更容易"想象"不存在的参数 |

---

## 2. OpenClaw 最简工具集法则

> **法则：用最少的工具，完成最多的任务。**

OpenClaw 的核心工具集只有 **5 个**：

```
┌─────────────────────────────────────────────────────────┐
│                OpenClaw 核心工具集                       │
│                                                          │
│  1. bash          执行 shell 命令（终极瑞士军刀）         │
│  2. read_file     读取文件内容                           │
│  3. edit_file     精准编辑文件（见第 07 节）              │
│  4. write_file    创建/覆盖文件                          │
│  5. list_dir      列出目录结构                           │
│                                                          │
└─────────────────────────────────────────────────────────┘
```

**为什么这 5 个就够了？**

- `bash` 可以执行任何 shell 命令，包括 curl/wget（替代 http 工具）、grep（替代 search 工具）、python 脚本（替代专用工具）
- `read_file` + `edit_file` + `write_file` 覆盖所有文件操作
- `list_dir` 帮助 Agent 理解项目结构

## 3. YOLO 执行哲学

### 什么是 YOLO？

YOLO（You Only Live Once）在这里的含义是：

> **信任 LLM 的判断，给它直接执行的权力，而不是在每一步都添加"你确定吗？"的保护层。**

### YOLO vs 过度保护

```python
# ❌ 过度保护：每步都要确认
async def cautious_bash(command: str, confirmed: bool = False) -> str:
    if not confirmed:
        return f"即将执行: {command}\n请传入 confirmed=True 来确认执行"
    return subprocess.run(command, ...).stdout

# ✓ YOLO 执行：直接运行
async def bash(command: str) -> str:
    """执行 shell 命令"""
    return subprocess.run(command, shell=True, capture_output=True, text=True).stdout
```

### YOLO 不等于无边界

YOLO 哲学不是"什么都做"，而是：

**内层：YOLO（信任，直接执行）**  
**外层：Middleware 拦截高危操作（见第 16 节）**

```
工具调用请求
    ↓
[Middleware 层] ← 只拦截真正危险的操作（rm -rf、生产数据库写入等）
    ↓
[YOLO 执行层] ← 其余操作直接执行，不问多余的问题
    ↓
返回结果
```

## 4. 工具设计七原则

### 原则一：单一职责

```python
# ❌ 多合一工具（模糊）
async def file_operation(path: str, action: str, content: str = None):
    if action == "read": ...
    elif action == "write": ...
    elif action == "delete": ...

# ✓ 独立工具（清晰）
async def read_file(path: str) -> str: ...
async def write_file(path: str, content: str) -> str: ...
async def delete_file(path: str) -> str: ...
```

### 原则二：描述要精准，不要模糊

```python
# ❌ 模糊描述
@registry.tool(description="处理文件")
async def process_file(path: str): ...

# ✓ 精准描述
@registry.tool(description="读取文件的完整内容并以字符串形式返回。适用于读取代码、配置文件、日志等文本文件。")
async def read_file(path: str) -> str: ...
```

### 原则三：失败要明确报错

```python
# ❌ 静默失败
async def read_file(path: str) -> str:
    try:
        with open(path) as f:
            return f.read()
    except:
        return ""  # Agent 不知道发生了什么

# ✓ 明确报错
async def read_file(path: str) -> str:
    try:
        with open(path, "r", encoding="utf-8") as f:
            return f.read()
    except FileNotFoundError:
        return f"错误：文件不存在 - {path}"
    except PermissionError:
        return f"错误：无读取权限 - {path}"
    except UnicodeDecodeError:
        return f"错误：文件不是文本格式（可能是二进制文件）- {path}"
```

### 原则四：输出要简洁，避免噪声

```python
# ❌ 输出噪声太多
async def bash(command: str) -> str:
    result = subprocess.run(command, ...)
    return f"""
    命令: {command}
    退出码: {result.returncode}
    标准输出:
    ========
    {result.stdout}
    ========
    标准错误:
    ========
    {result.stderr}
    ========
    执行时间: {elapsed}ms
    """  # 大量格式化噪声消耗 Token

# ✓ 简洁输出
async def bash(command: str) -> str:
    result = subprocess.run(command, shell=True, capture_output=True, text=True)
    output = result.stdout.strip()
    if result.returncode != 0 and result.stderr:
        output += f"\n[Error]: {result.stderr.strip()}"
    return output or "(无输出)"
```

### 原则五：幂等性（能重复执行）

```python
# ✓ 幂等写入（多次执行结果相同）
async def write_file(path: str, content: str) -> str:
    os.makedirs(os.path.dirname(path), exist_ok=True)
    with open(path, "w", encoding="utf-8") as f:
        f.write(content)
    return f"已写入: {path}"

# ❌ 非幂等（每次执行结果不同）
async def append_to_log(message: str) -> str:
    with open("log.txt", "a") as f:
        f.write(message + "\n")  # 重试时会重复追加
```

### 原则六：参数越少越好

```python
# ❌ 参数过多，LLM 容易出错
async def search_files(
    path: str,
    pattern: str,
    recursive: bool = True,
    case_sensitive: bool = False,
    max_results: int = 100,
    file_type: str = None,
    exclude_dirs: list = None,
) -> str: ...

# ✓ 简单参数，复杂逻辑内部处理
async def bash(command: str) -> str:
    """直接用 find/grep 命令完成复杂搜索"""
    ...
```

### 原则七：工具名称要自解释

```python
# ❌ 晦涩的工具名
registry.register("fn_execute", ...)
registry.register("proc_txt", ...)

# ✓ 自解释的工具名
registry.register("bash", ...)
registry.register("read_file", ...)
registry.register("write_file", ...)
```

## 5. 工具集扩展的正确时机

什么时候才应该添加新工具？

**✓ 应该添加新工具：**
- bash 无法完成（如：浏览器自动化、数据库 ORM 操作）
- 有外部服务 API 需要集成（如：飞书、Slack）
- 性能敏感的操作（如：大文件搜索，用原生实现比 bash 快 10 倍）

**✗ 不应该添加新工具：**
- bash 已经能做到的事情
- 只是"看起来更专业"
- 为了演示而不是生产需要

## 📝 本节小结

| 哲学 | 核心要点 |
|---|---|
| 最简工具集 | 5 个核心工具（bash + 4 个文件操作）覆盖大多数场景 |
| YOLO 哲学 | 信任 LLM，减少不必要保护层，通过 Middleware 拦截真正危险操作 |
| 工具设计原则 | 单一职责、精准描述、明确报错、简洁输出、幂等性 |
