# 01 架构演进：从 Framework 到 Harness，Agent 需要怎样的底层支撑？

## 🎯 本节目标

理解 Agent Framework 与 Agent Harness 的本质区别，掌握为什么现代生产级 Agent 需要更轻量、更可控的底层支撑。

---

## 1. Framework 的诞生与困境

### 1.1 为什么会有框架？

2023 年初，当 ChatGPT 引爆 AI 浪潮，开发者们迫切需要一种方式来构建"能自主完成任务的程序"。  
LangChain 率先提出了一套抽象：

```
Chain → Agent → Tool → Memory → Retriever → ...
```

这套抽象在**演示阶段**极为好用——几行代码就能跑起来一个"智能助手"。

### 1.2 框架的三大原罪

随着落地场景增多，Framework 的问题逐渐暴露：

| 问题 | 具体表现 |
|---|---|
| **黑盒化** | Prompt 被多层封装，不知道发给模型的是什么 |
| **抽象泄漏** | 底层 API 升级，框架却没跟上，Bug 难以追踪 |
| **过度设计** | 为了覆盖所有场景，核心流程被无数中间件包裹 |

**真实案例：** 某团队用 LangChain 构建 RAG Agent，线上突然出现"幻觉"。排查三天后发现：是 LangChain 内部的 Prompt 模板拼接顺序在某次版本更新后悄悄改变了，导致 System Prompt 和 User Message 的顺序颠倒。

---

## 2. 从框架到 Harness：思维转变

### 2.1 什么是 Harness？

**Harness** 本意是"马具"或"线束"——将各个组件绑在一起，但每个组件本身保持独立可替换。

在 Agent 领域，Harness 是：

> **一个最小化、可观测、可组合的运行时，专门为驱动 LLM Agent 而设计。**

它不替你做决策，只负责：
1. 驱动 LLM 的推理循环
2. 分发工具调用
3. 管理上下文窗口
4. 记录可观测数据

### 2.2 Framework vs Harness 对比

```
Framework 的思路：
  "我给你一个完整的解决方案，你只需要配置"
  
  用户代码 → Framework 抽象层 → LLM API
             ↑
         黑盒区域（难以调试）

Harness 的思路：
  "我给你一个最小运行时，你保留完全控制权"
  
  用户代码 ←→ Harness Core ←→ LLM API
              ↑
          全链路透明
```

### 2.3 架构演进路线图

```
阶段一：单次调用
  User → LLM → Response
  
阶段二：有工具调用
  User → LLM → Tool Call → Tool Result → LLM → Response

阶段三：ReAct Loop（框架时代）
  User → [框架管理循环] → Response
  问题：框架成了黑盒
  
阶段四：Harness（当前最优实践）
  User → [透明 Main Loop] → Response
  每一步都可观测、可干预
```

各模块职责：
- **Main Loop**：驱动 ReAct 循环，是 Harness 的心脏（第 02 节详讲）
- **Provider Adapter**：抽象 LLM 接口，支持 Claude/OpenAI 等（第 04 节详讲）
- **Tool Registry**：工具注册与分发（第 05 节详讲）
- **Context Manager**：上下文窗口管理（第 11-12 节详讲）
- **Middleware Chain**：拦截与审批（第 16 节详讲）

## 4. 核心设计原则

### 原则一：最小必要复杂度（Minimum Necessary Complexity）

> 只实现 Agent 实际需要的功能，不为"以后可能用到"而提前抽象。

**反例：** 在只需要调用 5 个工具的场景下，构建支持"插件市场"的动态工具系统。

**正例：** 一个能注册工具、按名称分发、返回结果的简单字典就够了。

### 原则二：全链路可观测（Full Observability）

> 系统中每一次 LLM 调用、每一次工具执行，都应该有日志可查。

```python
# 好的实践：每次调用都记录
async def call_llm(messages, tools):
    start = time.time()
    response = await provider.complete(messages, tools)
    logger.info(f"LLM call: {time.time()-start:.2f}s, tokens={response.usage}")
    return response
```

### 原则三：YOLO 哲学（信任模型，减少保护层）

> 不要在 Agent 和工具之间加太多"安全检查"——它们往往只是增加延迟和复杂度，却无法真正防止问题。

真正的安全来自：设计清晰的工具接口 + Middleware 层的关键操作拦截。

## 5. 代码入门：最简 Harness 骨架

```python
# harness/core.py - OpenClaw 核心骨架

import asyncio
from typing import Any

class AgentHarness:
    """OpenClaw 最简 Harness 骨架"""
    
    def __init__(self, provider, tool_registry, max_turns=50):
        self.provider = provider       # LLM Provider 适配器
        self.tools = tool_registry     # 工具注册表
        self.max_turns = max_turns     # 防止无限循环
        self.messages = []             # 对话历史
    
    async def run(self, user_message: str) -> str:
        """驱动一次完整的 Agent 任务"""
        self.messages.append({"role": "user", "content": user_message})
        
        for turn in range(self.max_turns):
            # 1. 调用 LLM
            response = await self.provider.complete(
                messages=self.messages,
                tools=self.tools.get_definitions()
            )
            
            # 2. 检查是否需要调用工具
            if not response.tool_calls:
                # 没有工具调用 → 任务完成
                return response.content
            
            # 3. 执行工具调用
            tool_results = await self._execute_tools(response.tool_calls)
            
            # 4. 将结果追加到消息历史，继续循环
            self.messages.append(response.to_message())
            self.messages.extend(tool_results)
        
        raise RuntimeError(f"超过最大轮次 {self.max_turns}")
    
    async def _execute_tools(self, tool_calls) -> list:
        """执行工具调用列表"""
        results = []
        for call in tool_calls:
            result = await self.tools.dispatch(call.name, call.args)
            results.append({
                "role": "tool",
                "tool_call_id": call.id,
                "content": str(result)
            })
        return results
```

这 40 行代码就是 Harness 的核心骨架。后续章节将逐步填充每个细节。

## 📝 本节小结

| 概念 | 要点 |
|---|---|
| Framework | 完整解决方案，黑盒化，难以调试 |
| Harness | 最小运行时，全链路透明，可组合 |
| OpenClaw | 遵循"最小必要复杂度 + 全链路可观测 + YOLO 哲学" |
| 核心架构 | Main Loop + Provider + Tool Registry + Context Manager |
